Application

Style guide is about consistency. In some cases, inconsistence is justfied, e.g. not to break backwards compatibility.

Code lay-out

Indentation

  • use 4 spaces; do not use tabs.
  • continuation lines should be indented; use 2 or 4 spaces.
  • closing brace/bracket/parenthesis in multi-line construct goes in its own line.
  • closing brace/bracket/parenthesis may align with first non-blank character in previous line or with first non-blank character in the line that starts the multi-line construct.
Indentation examples:
 
my_list = [
    1, 2, 3,
    4, 5, 6,
    ]
 
result = some_function_that_takes_arguments(
    'a', 'b', 'c',
    'd', 'e', 'f',
    )
 
my_list = [
    1, 2, 3,
    4, 5, 6,
]
 
result = some_function_that_takes_arguments(
    'a', 'b', 'c',
    'd', 'e', 'f',
)

Maximum line length

  • 79 characters for code, 72 characters for comments and docstrings.
  • implicit continuation: expressions within brace/brackets/parenthesis may span over several lines; this is the preferred method for wrapping long lines, enclosing them within braces/brackets/parenthesis.
  • use continuation character "/" when implicit continuation is not possible, e.g. multiple with statements, assert statements.
  • binary operators: preferred place to break is after binary operator.

Blank lines

  • separate top-level function and class statements with two blank lines.
  • separate method definitions inside a class with one blank line.
  • within functions, blank lines may be used (sparingly) to separate logical sections.

Source file encoding

  • use UTF-8.
  • using \x, \u, \U, or \N escapes is the preferred way to include non-ASCII data in string literals.

Imports

  • place each import statement on a separate line.
  • this is correct, though:
       from subprocess import Popen, PIPE
  • do not use wildcard "*"  imports.
  • absolute imports are recommended, e.g.
       from mypkg.sibling import example

Whitespace in expressions and statements

  • Avoid extraneous whitespace:
    • immediately inside parenthesis/brackets/braces.
    • immediately before comma (,), semicolon (;) or colon (:).
    • immediately before the open parenthesis that starts the argument list of a function, an indexing or slicing.
  • Surround assignment operators with a single space:   a == b
  • Do not use spaces around = when used in keyword arguments or defaul value in parameters.
This is correct:
    def fn(par1, par2='alpha'):
        self.fld1 = par1 * (2 - math.pi)

Other recommendations

  • avoid compound statements (more than one statement in a single line).
  • do not put an if/for/while body in the same line of if/for/while keyword, except perhaps if body is a single line.
  • do not put try/except bodies in same line as try or except keywords.
  • do not call several functions in a line.
This is correct:
   if val >= 15:
        self.fld1 = afun1()
        afun2(self.fld1)
    else:
        self.fld1 = None

Comments

  • comments that contradict the code are worse than no comments. Keep comments up to date!
  • comments should be complete sentences, start with a capital, and end with a period.
  • always keep case of identifiers!
Block comments:
  • block comments apply to a block of code that follows.
  • use # and a single space to start each comment line.
Inline comments:
  • an inline comment is a comment included in a statement, from # to the end of the line.
  • use inline comments sparingly.

Naming Conventions

  • Names in the public part of an API should reflect usage rather than implementation.
  • conventions for underscores:
    • _single_leading_underscore: for internal objects, from M import * does not import them.
    • single_trailing_underscore: to avoid conflict with Python keyword, e.g. class_.
    • __double_leading_underscore: when naming a class attribute, invokes name mangling: inside class FooBar, __boo becomes _FooBar__boo.
    • __double_leading_and_trailing_underscore__: objects or attributes that live in user-controlled namespaces, e.g. __init__, __import__, __file__. Never invent such names, use as documented.
  • never use characters l, O, I (el, oh, eye) as single character variable names; in some fonts they are indistinguishable from numerals zero and one.
Package and Module names
  • modules should have short, all-lowercase names. Underscores can be used for readability.
  • packates should have short, all-lowercase names. Underscores are discouraged.
Class names, Exception names
  • Class names should use CapWords convention (CamelCase convention).
  • Exceptions are classes, use class name convention plus Error, like this: InvalidCodeError.
Global variable names
  • for use only inside a module.
  • use __all__ mechanism to prevent their import, or name them starting with an underscore, like this: _a_global_variable.
  • use function name convention (lowercase, and underscores).
Functions and methods, arguments
  • function names should be lowercase, with words separated by underscores to improve readability.
  • mixedCase allowed only for backwards compatibility.
  • always use self for first argument to instance methods.
  • always use cls as first argument to class methods.
  • in function arguments, avoid clashes with keywords by using a trailing underscore, like this: class_.
  • for method names and instance variables, follow function naming rules: lowercase, and underscores.
  • use one leading underscore for non-public methods and non-public instance variables.
Constants
  • define constants at module level.
  • name constants in all capital letters with underscores separating words.
